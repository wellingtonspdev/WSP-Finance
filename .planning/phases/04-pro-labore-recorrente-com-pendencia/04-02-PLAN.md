<threat_model>
## Trust Boundaries

| Boundary | Description |
|---|---|
| Browser/API client to controller | Authenticated request DTOs cross into backend validation. |
| Controller to service | Route/header context is insufficient for cross-workspace authorization. |
| Service to Prisma | Financial schedule and pending state persists under tenant-sensitive rules. |
| Service to BridgeService | Manual confirmation crosses into balance/ledger mutation. |
| Cron to service | Background job creates pending rows without user interaction. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|---|---|---|---|---|
| T-P4-02-01 | Elevation of privilege | OWNER checks | mitigate | Query both workspace memberships as OWNER before every mutation and confirmation. |
| T-P4-02-02 | Tampering | Cron generation | mitigate | Tests assert no BridgeService call, no Transaction creation, and no balance mutation from cron. |
| T-P4-02-03 | Tampering | Confirmation idempotency | mitigate | Persist a durable pending bridge id/idempotency key, claim/lock the pending row before bridge execution, pass the key into BridgeService, and complete the pending only after idempotent bridge success. |
| T-P4-02-06 | Tampering | Crash/retry boundary | mitigate | BridgeService uses deterministic FITIDs from the caller key and existing unique constraints so retry after bridge success cannot create duplicate transfer legs or mutate balances twice. |
| T-P4-02-04 | Information disclosure | lastError | mitigate | Store only safe normalized error labels/messages; never raw payloads, PII, OCR, or tokens. |
| T-P4-02-05 | Denial of service | Cron duplicate/missed runs | mitigate | Use unique schedule plus competence and process due schedules up to today. |
| T-P4-02-SC | Tampering | npm/pip/cargo installs | accept | No package installation is planned. |
</threat_model>
