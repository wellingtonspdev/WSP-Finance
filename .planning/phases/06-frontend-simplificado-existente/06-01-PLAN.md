# 06-01 Plan - Frontend Simplificado Existente

## Objective

Align existing frontend flows with backend contracts from Phases 2, 3, and 5. Users must not choose accounts in primary transaction, bridge/pro-labore, or Telegram flows. Frontend payloads must not send public `accountId`, `fromAccountId`, or `toAccountId` in those flows.

## Scope

### In

- Simplify `TransactionModal` existing UI.
- Remove common transaction account selector.
- Remove bridge origin/destination account selectors.
- Keep bridge destination workspace selector.
- Stop sending `accountId` from common transaction submit.
- Keep bridge payload limited to workspaces, amount, description, and date.
- Adjust transaction request/form types to remove public account fields.
- Update Telegram config copy so account selection is not promised.
- Adjust transaction list/item display so account is not primary and tax/net labels are hidden when automatic tax data is null.
- Add focused frontend tests.

### Out

- Backend changes.
- Prisma migrations.
- Recurrence, pending pro-labore, cron.
- New broad UI screens.
- Account admin/accountant/import APIs and screens unless a shared type breaks the primary flow.

## Implementation Steps

### 1. Add Payload Sanitizer Tests First

Create a focused test around transaction payload assembly before changing UI.

Preferred shape:
- Extract pure helpers from `useTransactionMutation` into a local module, for example:
  - `buildTransactionPayload(data)`
  - `buildBridgePayload(data, activeWorkspaceId)`
- Keep helper file inside `frontend/src/features/transactions/`.

Required test scenarios:
- Common transaction payload does not contain `accountId`.
- Marketplace payload does not contain `accountId` and preserves `grossAmount`, `marketplaceFee`, `shippingCost`, and `productCost`.
- Bridge payload contains `fromWorkspaceId`, `toWorkspaceId`, `amount`, `description`, `date`.
- Bridge payload does not contain `accountId`, `fromAccountId`, or `toAccountId`.

### 2. Simplify Transaction Types

Update frontend request/form schemas in `frontend/src/features/transactions/types/index.ts`.

Rules:
- Remove `accountId` and `toAccountId` from `transactionFormSchema`.
- Remove `accountId` and `toAccountId` from `transactionPayloadSchema`.
- Keep `Transaction.accountId` response field.
- Keep `TransactionFilters.accountId` only if existing filters rely on it.
- Keep category validation for non-bridge transactions.
- Keep `toWorkspaceId` validation for bridge.

### 3. Simplify Transaction Modal

Update `frontend/src/features/transactions/components/TransactionModal.tsx`.

Rules:
- Remove account-related defaults from `useForm`.
- Remove `useAccounts` and `useAccountsByWorkspace` usage if no longer needed.
- Remove common transaction "Conta Bancaria" selector.
- For non-bridge rows, render category as the full-width selector or otherwise keep layout stable.
- Remove bridge "Conta de Origem" and "Conta que vai Receber" selectors.
- Keep bridge "Ambiente de Destino" selector.
- Keep amount, description, date, status, category, attachment, and marketplace fields.
- Update bridge empty-state/toast copy to say "Selecione o workspace de destino", not "Preencha todas as contas".

### 4. Simplify Submit Payloads

Update `frontend/src/features/transactions/hooks/useTransactionMutation.ts`.

Rules:
- Use helper(s) from Step 1.
- Remove the line that conditionally adds `sanitizedPayload.accountId`.
- Ensure bridge submit calls `createBridge` with only:
  - `fromWorkspaceId`
  - `toWorkspaceId`
  - `amount`
  - `description`
  - `date`
- Preserve upload/R2 behavior before common transaction creation.
- Preserve backend error parsing and existing success callbacks.

### 5. Telegram Config Copy

Update `frontend/src/features/workspaces/routes/TelegramConfigPage.tsx`.

Rules:
- Do not add any account selector.
- Remove wording saying the bot asks for "conta".
- Keep user account/link wording only where it means Telegram/WSP account, not bank account.
- Do not remove `TelegramDestination.accountId` response type.

### 6. Transaction Display

Update `frontend/src/features/transactions/components/TransactionAccordionItem.tsx`.

Rules:
- Primary row should show category and description, not "Conta: ...".
- Account can be omitted from the primary row.
- Do not remove `transaction.accountId` from the response model.
- For marketplace:
  - Keep gross/fee/shipping/product rows when values exist.
  - Show tax row only when `taxAmount` is non-null and greater than zero.
  - Use "Total" instead of "Total Liquido" when `netValue` is null.
  - Use "Total Liquido" only when `netValue` is not null.
- For non-marketplace, keep simple total display.

## Test Plan

Run tests from `frontend`.

Required automated tests:
- Payload helper excludes `accountId` for common transaction.
- Payload helper excludes `accountId` and preserves marketplace fields.
- Bridge payload helper excludes `fromAccountId` and `toAccountId`.
- `TransactionModal` renders without visible account fields.
- `TelegramConfigPage` renders without account selector/copy promising account selection.
- `TransactionAccordionItem` does not show tax/net wording when `taxAmount` and `netValue` are null.

Required validation commands:

```powershell
cd frontend
pnpm test
pnpm run build
cd ..
git diff --check
```

## Acceptance Criteria

- User does not see account selection in primary manual transaction flow.
- User does not see origin/destination account selection in manual bridge/pro-labore flow.
- Frontend transaction submit does not send `accountId`.
- Frontend bridge submit does not send `accountId`, `fromAccountId`, or `toAccountId`.
- Telegram config does not expose bank account selection.
- Transaction item/list does not emphasize account as primary information.
- Tax/net UI does not present automatic tax data when backend returns null.
- Backend, migrations, recurrence, pending flows, cron, and broad dashboard UI remain untouched.

## Plan Review

Inline review found no P0/P1/P2 blockers after these constraints:

- Keep account fields in response/read types.
- Remove account only from public primary-flow requests and visible selectors.
- Add tests around pure payload helpers to avoid brittle hook-only tests.
- Preserve bridge API because it already matches backend Phase 3.
