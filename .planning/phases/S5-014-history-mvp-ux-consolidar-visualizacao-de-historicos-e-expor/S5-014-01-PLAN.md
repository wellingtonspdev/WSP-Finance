# S5-014-01 Plan - Backend Export History API

## Objective

Add a tenant-safe export-history listing endpoint over existing `ExportArchive` records. The endpoint must expose only safe metadata needed by the frontend and must reuse existing workspace/RBAC patterns without changing export generation or download behavior.

## Scope

### In

- Add a service method to list `ExportArchive` rows by workspace.
- Add a controller for `GET /workspaces/:workspaceId/exports`.
- Register the route with `AuthMiddleware`, `WorkspaceRouteParamGuard`, `WorkspaceMiddleware`, and `RbacMiddleware('ACCOUNTANT')`.
- Return safe metadata: id, period, layout, target system, generated date, creator summary, status, file name, sha256, `recordCount`, `warningsCount`, content type.
- Preserve existing download route and `ExportArchiveService.getDownloadUrl()`.
- Add focused service and route tests for tenant isolation, RBAC, and safe response shape.

### Out

- No Prisma migration unless the current schema cannot satisfy the DTO.
- No `ExportService` generation change.
- No Domínio layout, encoding, or validation change.
- No new storage provider behavior.
- No second download endpoint.
- No frontend implementation; covered by S5-014-02.

## Must Haves

### Truths

- D-06: Backend must provide period, layout, generation date, status, `recordCount`, `warningsCount`, and download-capable archive id.
- D-07: Backend must provide responsible user and hash for expandable details.
- D-08: Backend returns the full hash to the authorized client, but never storage internals.
- D-09: Route permission must align with export/download: OWNER and ACCOUNTANT pass via `RbacMiddleware('ACCOUNTANT')`; EDITOR/VIEWER fail.
- D-10: Route must return `403` through existing middleware for unauthorized workspace access.
- D-12: Listing endpoint is `GET /workspaces/:workspaceId/exports`.

### Safety Invariants

- Use `prisma` from `backend/src/lib/prisma.ts`, not `sysPrisma`.
- Query must be scoped by `workspaceId`.
- DTO must not contain `objectKey`, bucket, buffer, base64, raw text, file contents, path, provider errors, or storage config.
- Existing `GET /workspaces/:workspaceId/exports/:archiveId/download` behavior must remain unchanged.
- RLS/RBAC/LGPD remain first-class acceptance criteria.

## Implementation Steps

### 1. Add RED Service Tests

Create `backend/tests/services/ExportArchiveService.history.test.ts`.

Required scenarios:
- Creates at least two workspaces and users with `managementClient`.
- Inserts multiple `ExportArchive` rows across two workspaces.
- `listByWorkspace(wsAId)` returns only workspace A rows.
- Results are sorted by `createdAt desc`.
- Returned records include creator summary and hash metadata.
- Returned records do not expose `objectKey`, bucket, buffer, base64, path, or raw file content.
- Application/RLS proof: under tenant B context, `applicationClient.exportArchive.findMany({ where: { id: archiveA.id } })` returns zero.

Use existing helpers from `backend/src/test/prisma-test-clients.ts` and current ExportArchive tests where possible.

### 2. Add RED Controller/Route Tests

Create `backend/tests/controllers/ExportHistoryController.test.ts`, mirroring `ExportDownloadController.test.ts`.

Required scenarios:
- `GET /workspaces/:workspaceId/exports` returns `200` for OWNER.
- Returns `200` for ACCOUNTANT.
- Returns `403` for EDITOR.
- Returns `403` for VIEWER.
- Returns `401` without token.
- Returns `403` when `x-workspace-id` diverges from `:workspaceId`.
- Works when header is absent and route param is valid, matching download route behavior.
- Returns `400` for invalid route workspace id.
- Returns `403` for non-member.
- Response body never contains forbidden storage/internal keys.

Mock `ExportArchiveService.listByWorkspace` in controller tests to isolate route/middleware behavior.

### 3. Implement Service List Method

Update `backend/src/services/ExportArchiveService.ts`.

Suggested shape:

```ts
async listByWorkspace(workspaceId: number): Promise<ExportArchiveHistoryItem[]>
```

Rules:
- `where: { workspaceId }`
- `orderBy: { createdAt: 'desc' }`
- `include` or `select` only safe creator fields.
- Compute status as `AVAILABLE` for archived rows unless current model proves a better status source exists.
- Map `sha256` to a safe public field such as `hash` or keep `sha256` consistently with current backend naming.
- Do not return Prisma rows directly if they include `objectKey`.

### 4. Add Controller

Create `backend/src/controllers/ExportHistoryController.ts`.

Rules:
- Validate `req.params.workspaceId` as a positive integer.
- Verify `req.workspaceId` matches route workspace id defensively, like `ExportDownloadController`.
- Call `ExportArchiveService.listByWorkspace(routeWorkspaceId)`.
- Return `200` with `{ data: items }`.
- Let expected `AppError`/middleware errors flow through existing error handling.

### 5. Register Route

Update `backend/src/routes.ts`.

Add route near the existing export download route:

```ts
router.get(
  '/workspaces/:workspaceId/exports',
  AuthMiddleware,
  WorkspaceRouteParamGuard,
  WorkspaceMiddleware,
  RbacMiddleware('ACCOUNTANT'),
  (req, res) => exportHistoryController.list(req, res)
);
```

Keep the existing download route unchanged.

## Acceptance Criteria

- `GET /workspaces/:workspaceId/exports` exists and is protected by auth, route-param guard, workspace middleware, and RBAC.
- OWNER and ACCOUNTANT can list export history for authorized BUSINESS workspaces.
- EDITOR, VIEWER, non-member, missing token, invalid workspace id, and divergent header cases are blocked.
- Cross-tenant records are not returned.
- Response includes all fields required for S5-014 frontend cards and expandable details.
- Response excludes storage internals and raw sensitive data.
- Existing download endpoint remains covered and unchanged.
