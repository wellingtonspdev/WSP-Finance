# Phase 3 Plan

## Goal

Implement manual bridge/pro-labore transfers where the client sends workspaces only and the backend resolves the default source and target accounts.

## Non-Goals

- No recurrence.
- No pending pro-labore.
- No cron.
- No new recurring UI.
- No account auto-creation.
- No broad middleware refactor.
- No schema migration.
- No Telegram/OCR changes.

## Files To Change

Primary:

- `backend/src/controllers/BridgeController.ts`
- `backend/src/services/BridgeService.ts`
- `backend/tests/services/BridgeService.balance-audit.test.ts`

Likely:

- `frontend/src/features/workspaces/api/executeBridgeTransfer.ts`
- `backend/src/swagger-output.json`

Only if needed:

- `backend/tests/controllers/BridgeController.test.ts`

## TDD Steps

### Step 1 - Service DTO without account IDs

Update `BridgeService.balance-audit.test.ts` first:

- remove `fromAccountId` and `toAccountId` from the DTO passed to `executeTransfer`;
- mock `AccountRepository.findDefaultByWorkspace`;
- assert it is called with `fromWorkspaceId` plus source workspace type and `toWorkspaceId` plus target workspace type.

### Step 2 - Resolve default accounts in service

Change `BridgeTransferDTO` by removing `fromAccountId` and `toAccountId`.

In `BridgeService.executeTransfer`, after membership and closed-period validation, resolve:

```ts
fromAccount = await accountRepository.findDefaultByWorkspace(dto.fromWorkspaceId, fromMembership.workspace.type)
toAccount = await accountRepository.findDefaultByWorkspace(dto.toWorkspaceId, toMembership.workspace.type)
```

Throw controlled `AppError` if either account is missing.

### Step 3 - Preserve balance and transaction writes

Use `fromAccount.id` and `toAccount.id` for debit/credit transaction data and account balance updates.

Keep the existing single `prisma.$transaction` boundary.

### Step 4 - Preserve audit contract

Audit rows must keep:

- `fromAccount: fromAccount.id`
- `toAccount: toAccount.id`
- `oldState.accountId` matching the leg account;
- balance before/after snapshots;
- opposite deltas.

### Step 5 - Failure cases

Add service tests for:

1. Missing default source account: throws before `$transaction`.
2. Missing default target account: throws before `$transaction`.
3. Insufficient source balance: throws before `$transaction`.
4. Missing membership in one workspace: throws `AppError` 403 and does not resolve accounts.
5. Closed fiscal period: blocks before transaction writes.

### Step 6 - Controller contract

Update `BridgeController` Zod schema:

- remove `fromAccountId`;
- remove `toAccountId`;
- keep `fromWorkspaceId`, `toWorkspaceId`, `amount`, `description`, and `date`.

Keep same-workspace guard.

Review catch behavior so `AppError` statuses do not become 500.

### Step 7 - Frontend API type

Update `frontend/src/features/workspaces/api/executeBridgeTransfer.ts` to remove account IDs from `BridgeTransferDTO`.

Do not build new UI.

### Step 8 - Swagger

Regenerate/update Swagger only after backend contract is correct. `POST /bridge/transfer` must no longer require or document `fromAccountId` or `toAccountId`.

## Risks

P0: partial ledger write if transaction boundary is changed.

P1: wrong account resolution across workspaces; missing permission check before account resolution; audit loses resolved account IDs; closed-period guard returns 500 instead of 403.

P2: Swagger/frontend type drift; route uses only `AuthMiddleware`, so service validation remains critical.

P3: default account naming may not match UI wording. Existing fallback mitigates via `CHECKING` then first account.

## Stop Criteria

Stop and ask before continuing if default account must be auto-created, service requires new `DefaultAccountService`, schema/migration is needed, tests require broad route/middleware redesign, recurring pro-labore appears necessary, or Telegram/OCR files need changes.

## Execution Prompt

Implement Phase 3 only. Adjust manual bridge/pro-labore transfer so client no longer sends `fromAccountId` or `toAccountId`. Use existing `AccountRepository.findDefaultByWorkspace(workspaceId, workspace.type)` directly in `BridgeService` to resolve source and target accounts. Preserve membership checks, same-workspace guard, closed fiscal-period guard, insufficient-balance guard, one Prisma transaction, debit/credit transaction creation, account balance updates, and audit snapshots with resolved `fromAccount` and `toAccount`. Do not implement recurrence, pending pro-labore, cron, schema migration, or new UI. Update focused bridge tests and minimal frontend API type/Swagger only as needed. Validate with TypeScript and `BridgeService.balance-audit.test.ts`.
